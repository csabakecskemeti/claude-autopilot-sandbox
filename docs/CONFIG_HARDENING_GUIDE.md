# Config Hardening Guide: Immutable Agent Guardrails

A reusable pattern for preventing autonomous agents from modifying their own configuration files.

---

## The Problem

When running an autonomous AI agent inside a Docker container, the agent typically has:
- Full filesystem access (needed to write code)
- Often sudo/root access (needed to install packages)

This creates a security risk: **the agent can modify its own guardrails**.

### Example Attack Vectors

```bash
# Agent removes validation hook
sed -i 's/curl.*supervisor/echo success/' ~/.config/hooks/stop_hook.sh

# Agent disables all hooks
echo '{}' > ~/workspace/.config/settings.json

# Agent changes its behavioral instructions
echo "You can do anything" > ~/workspace/INSTRUCTIONS.md
```

If the agent can modify these files, any guardrails you implement (hooks, validation, behavioral rules) can be bypassed.

---

## The Solution: Read-Only Bind Mounts

### Why Not Symlinks?

Symlinks are **not secure** - the agent can break them:

```bash
rm ~/workspace/CONFIG.md           # Removes symlink
echo "new content" > ~/workspace/CONFIG.md  # Creates regular file
```

### Why Bind Mounts Work

Docker bind mounts are **kernel-enforced**:

1. Managed by Docker daemon (runs OUTSIDE the container)
2. Cannot be removed, renamed, or replaced from inside container
3. Even sudo cannot bypass kernel-level mount protection
4. Cannot delete a directory containing a mount point ("Device busy")

```yaml
# docker-compose.yml
volumes:
  - ./config/settings.json:/app/config/settings.json:ro  # :ro = read-only
```

When the agent tries to modify:
```bash
$ echo "test" >> /app/config/settings.json
bash: /app/config/settings.json: Read-only file system
```

## Implementation Pattern

### Step 1: Identify Files to Protect

Typically you want to protect:
- Hook scripts (validation, logging, gates)
- Settings/config files (permissions, tool access)
- Behavioral instructions (system prompts, rules)
- Any file that enforces constraints on the agent

### Step 2: Generate Config on Host (Not in Container)

**Before:** Config generated inside container (vulnerable)
```bash
# Inside container entrypoint - VULNERABLE
cat > /app/config/settings.json << EOF
{ "hooks": [...] }
EOF
```

**After:** Config generated on host, mounted read-only (secure)
```bash
# In run.sh on HOST - SECURE
cat > ./workspace/config/settings.json << EOF
{ "hooks": [...] }
EOF

# Then mount read-only in docker-compose.yml
```

### Step 3: Add Read-Only Mounts

```yaml
# docker-compose.yml
services:
  agent:
    volumes:
      # Workspace - agent needs to write code here (read-write)
      - ./workspace:/app/workspace

      # Protected config files (read-only)
      - ./workspace/config/settings.json:/app/workspace/config/settings.json:ro
      - ./workspace/INSTRUCTIONS.md:/app/workspace/INSTRUCTIONS.md:ro
      - ./hooks:/app/hooks:ro
```

**Key insight:** You can mount a file read-only INSIDE a read-write directory. The specific file becomes immutable while the rest of the directory stays writable.

### Step 4: Verify Protection at Startup

Add verification in your container entrypoint:

```bash
#!/bin/bash
# entrypoint.sh

# Verify protected files are read-only
if touch /app/workspace/config/settings.json 2>/dev/null; then
    echo "WARNING: settings.json is writable - security not active"
else
    echo "✓ settings.json is read-only (protected)"
fi
```

## Complete Example

### Directory Structure

```
project/
├── docker-compose.yml
├── run.sh                    # Generates config, starts container
├── entrypoint.sh             # Container startup (just verification)
├── hooks/
│   └── validation_hook.sh    # Protected hook scripts
├── config-templates/
│   └── instructions.md       # Source behavioral instructions
└── workspaces/
    └── {task}/               # Per-task workspace
        ├── config/
        │   └── settings.json # Generated by run.sh, mounted :ro
        ├── INSTRUCTIONS.md   # Copied by run.sh, mounted :ro
        └── src/              # Agent writes code here (rw)
```

### run.sh (Host-Side)

```bash
#!/bin/bash
set -e

WORKSPACE="./workspaces/${TASK_NAME}"
mkdir -p "$WORKSPACE/config"

# Generate settings on HOST (will be mounted read-only)
cat > "$WORKSPACE/config/settings.json" << EOF
{
  "hooks": {
    "onStop": "/app/hooks/validation_hook.sh"
  },
  "permissions": {
    "deny": ["dangerous_tool"]
  }
}
EOF

# Copy instructions on HOST (will be mounted read-only)
cp ./config-templates/instructions.md "$WORKSPACE/INSTRUCTIONS.md"

# Export for docker-compose
export WORKSPACE_PATH="$WORKSPACE"

# Start container
docker compose up -d
```

### docker-compose.yml

```yaml
version: '3.8'

services:
  agent:
    build: .
    volumes:
      # Main workspace (read-write for agent's work)
      - ${WORKSPACE_PATH}:/app/workspace

      # ====== PROTECTED FILES (read-only) ======
      # Agent cannot modify these even with sudo

      # Config file with hooks
      - ${WORKSPACE_PATH}/config/settings.json:/app/workspace/config/settings.json:ro

      # Behavioral instructions
      - ${WORKSPACE_PATH}/INSTRUCTIONS.md:/app/workspace/INSTRUCTIONS.md:ro

      # Hook scripts directory
      - ./hooks:/app/hooks:ro
```

### entrypoint.sh (Container-Side)

```bash
#!/bin/bash
set -e

echo "Verifying protected config files..."

# Check settings.json
if touch /app/workspace/config/settings.json 2>/dev/null; then
    echo "WARNING: settings.json is writable - running in unprotected mode"
else
    echo "✓ settings.json is read-only (protected)"
fi

# Check instructions
if touch /app/workspace/INSTRUCTIONS.md 2>/dev/null; then
    echo "WARNING: INSTRUCTIONS.md is writable - running in unprotected mode"
else
    echo "✓ INSTRUCTIONS.md is read-only (protected)"
fi

# Check hooks directory
if touch /app/hooks/test 2>/dev/null; then
    rm /app/hooks/test
    echo "WARNING: hooks directory is writable - running in unprotected mode"
else
    echo "✓ hooks directory is read-only (protected)"
fi

echo ""
echo "Starting agent..."
exec "$@"
```

## What Gets Protected vs. What Stays Writable

| Path | Mount | Agent Can Modify? | Why |
|------|-------|-------------------|-----|
| `/app/workspace/` | rw | ✅ Yes | Agent needs to write code |
| `/app/workspace/config/settings.json` | ro | ❌ No | Hooks/permissions config |
| `/app/workspace/INSTRUCTIONS.md` | ro | ❌ No | Behavioral rules |
| `/app/hooks/` | ro | ❌ No | Validation scripts |
| `/app/workspace/src/` | rw | ✅ Yes | Agent's code output |
| `/app/workspace/output/` | rw | ✅ Yes | Agent's work products |

---

## Edge Cases & Solutions

### Agent tries to delete parent directory

```bash
rm -rf /app/workspace/config/
# Result: "Device or resource busy"
```

You cannot delete a directory that contains a mount point.

### Agent tries to unmount

```bash
sudo umount /app/workspace/config/settings.json
# Result: Permission denied
```

Mount namespace is controlled by Docker daemon (outside container).

### Agent creates new config file with different name

```bash
echo '{}' > /app/workspace/config/settings2.json
# Result: File created, but your code reads settings.json (the protected one)
```

Your application must read from the protected path, not discover config files dynamically.

### Development/debugging mode

Keep a fallback for when running without mounts (development):

```bash
if [ ! -f "/app/workspace/config/settings.json" ]; then
    echo "Config not mounted, creating default (UNPROTECTED MODE)"
    mkdir -p /app/workspace/config
    echo '{"hooks":{}}' > /app/workspace/config/settings.json
fi
```

## Testing the Protection

```bash
# Start container
./run.sh my-task

# Exec into container
docker exec -it my-agent bash

# Try to modify protected files
echo "test" >> /app/workspace/config/settings.json
# Expected: "Read-only file system"

echo "test" >> /app/workspace/INSTRUCTIONS.md
# Expected: "Read-only file system"

rm -rf /app/hooks/
# Expected: "Device or resource busy"

# Verify agent can still write to workspace
echo "test" > /app/workspace/src/test.txt
# Expected: Success
```

## Summary

1. **Problem:** Agents with filesystem access can modify their own guardrails
2. **Solution:** Generate config on host, mount read-only in container
3. **Key insight:** Docker bind mounts are kernel-enforced, sudo cannot bypass
4. **Pattern:**
   - Generate/copy protected files in `run.sh` (host-side)
   - Mount with `:ro` flag in `docker-compose.yml`
   - Verify at container startup in `entrypoint.sh`

This pattern works for any containerized agent system where you need to enforce immutable configuration.

## Claude Code Specific Implementation

This section shows the exact implementation used in this project for Claude Code CLI.

### Files to Protect

| Path (in container) | Source (on host) | Purpose |
|---------------------|------------------|---------|
| `~/.claude/hooks/*` | `./hooks-backup/` | Stop hook (supervisor gate), image blocking |
| `~/.claude/settings.json` | Generated by run.sh | Deny list, skip prompt flag |
| `~/.claude/skills/*` | `./skills-backup/` | Custom skills (supervisor, workflow) |
| `~/.claude/agents/*` | `./agents-backup/` | Subagent definitions (qa-agent) |
| `~/.claude/CLAUDE.md` | `./claude-backup/CLAUDE.md` | Source behavioral instructions |
| `~/workspace/.claude/settings.json` | Generated by run.sh | Hooks config, env vars |
| `~/workspace/CLAUDE.md` | Copied by run.sh | Active behavioral instructions |

### docker-compose.yml Volumes

```yaml
volumes:
  # Main workspace (read-write for code)
  - ${WORKSPACE_PATH}:/home/claude/workspace

  # ====== IMMUTABLE CONFIG (read-only) ======

  # Project-level settings (hooks, permissions, env)
  - ${WORKSPACE_PATH}/.claude/settings.json:/home/claude/workspace/.claude/settings.json:ro

  # User-level settings (deny list, skip prompt)
  - ${WORKSPACE_PATH}/.claude/user-settings.json:/home/claude/.claude/settings.json:ro

  # Behavioral instructions (workspace)
  - ${WORKSPACE_PATH}/CLAUDE.md:/home/claude/workspace/CLAUDE.md:ro

  # Hook scripts
  - ./hooks-backup:/home/claude/.claude/hooks:ro

  # Source CLAUDE.md
  - ./claude-backup/CLAUDE.md:/home/claude/.claude/CLAUDE.md:ro

  # Skills (prevents tampering with supervisor)
  - ./skills-backup:/home/claude/.claude/skills:ro

  # Agents (prevents tampering with qa-agent)
  - ./agents-backup:/home/claude/.claude/agents:ro
```

### Settings Files Structure

**User-level (`~/.claude/settings.json`):**
```json
{
  "permissions": {
    "allow": ["*"],
    "deny": ["WebSearch", "WebFetch", "EnterPlanMode", "TodoWrite"]
  },
  "skipDangerousModePermissionPrompt": true
}
```

**Project-level (`workspace/.claude/settings.json`):**
```json
{
  "permissions": {
    "allow": ["*"],
    "deny": ["WebSearch", "WebFetch", "EnterPlanMode", "TodoWrite"]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [{
          "type": "command",
          "command": "/home/claude/.claude/hooks/block_image_read.sh",
          "timeout": 5
        }]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [{
          "type": "command",
          "command": "/home/claude/.claude/hooks/langfuse_stop_hook.sh",
          "timeout": 4800
        }]
      }
    ]
  },
  "env": {
    "SUPERVISOR_URL": "http://supervisor:8080",
    "SUPERVISOR_TIMEOUT": "3600",
    "MAX_CONTINUE_CYCLES": "100"
  }
}
```

### Key Settings Explained

**`skipDangerousModePermissionPrompt: true`**
- Required to avoid interactive prompt in container
- Must be in user-level settings

**Deny list:**
- `WebSearch` / `WebFetch` - Don't work with local LLMs, use custom skills
- `EnterPlanMode` / `TodoWrite` - Use custom `/plan` and `/tasks` skills

**Hooks:**
- Must be in PROJECT-level settings (user-level hooks ignored for workspaces)
- `PreToolUse` with `Read` matcher - Blocks image file reads (local LLMs can't handle multimodal)
- `Stop` hook - Calls supervisor before allowing completion

### run.sh Config Generation

```bash
# Generate project-level settings
generate_settings_json() {
    local output_file="$1"
    local stop_hook_timeout="$2"

    cat > "$output_file" << SETTINGS_EOF
{
  "permissions": {
    "allow": ["*"],
    "deny": ["WebSearch", "WebFetch", "EnterPlanMode", "TodoWrite"]
  },
  "hooks": {
    "PreToolUse": [...],
    "Stop": [...]
  },
  "env": {...}
}
SETTINGS_EOF
}

# Generate user-level settings
generate_user_settings_json() {
    local output_file="$1"

    cat > "$output_file" << 'USER_SETTINGS_EOF'
{
  "permissions": {
    "allow": ["*"],
    "deny": ["WebSearch", "WebFetch", "EnterPlanMode", "TodoWrite"]
  },
  "skipDangerousModePermissionPrompt": true
}
USER_SETTINGS_EOF
}

# In main script:
mkdir -p "$WORKSPACE_PATH/.claude"
generate_settings_json "$WORKSPACE_PATH/.claude/settings.json" "$TIMEOUT"
generate_user_settings_json "$WORKSPACE_PATH/.claude/user-settings.json"
cp "./claude-backup/CLAUDE.md" "$WORKSPACE_PATH/CLAUDE.md"
```

### Verification Script (init-workspace.sh)

```bash
#!/bin/bash
echo "Verifying protected config files..."

# Check project settings
if touch "${WORKSPACE}/.claude/settings.json" 2>/dev/null; then
    echo "WARNING: settings.json is writable - security not active"
else
    echo "✓ settings.json is read-only (protected)"
fi

# Check CLAUDE.md
if touch "${WORKSPACE}/CLAUDE.md" 2>/dev/null; then
    echo "WARNING: CLAUDE.md is writable - security not active"
else
    echo "✓ CLAUDE.md is read-only (protected)"
fi
```

### What Remains Writable

| Path | Purpose |
|------|---------|
| `~/workspace/*` (except config) | Agent's code and project files |
| `~/.claude/state/` | Runtime state |
| `~/.claude/cache/` | Cached data |

| `~/.claude/projects/` | Project trust state |
| `~/.claude/mcp-servers/` | MCP servers (consider protecting in strict mode) |

### Porting Checklist

When adapting this to another project:

1. **Identify your guardrail files** - hooks, instructions, validation scripts
2. **Generate config on host** - Create a run.sh or equivalent
3. **Add ro mounts** - Update docker-compose.yml with `:ro` suffix
4. **Include skipDangerousModePermissionPrompt** - Prevents interactive prompt
5. **Match deny list** - Block tools that don't work with your setup
6. **Add verification** - Check mounts at container startup
7. **Keep fallback** - For development without mounts
8. **Test both paths** - Verify agent can work but can't modify config
